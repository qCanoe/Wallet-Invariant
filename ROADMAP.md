# Wallet Invariant 项目路线规划

> 版本：v0.1.0 → v1.0.0
> 更新日期：2026-03-16
> 当前状态：MVP 研究原型，核心框架完备，待真实场景验证与生产加固

---

## 当前状态评估

### 已完成（MVP 基线）
- [x] 四项执行不变式 (I1–I4) 完整实现
- [x] 交易分类器（启发式规则）
- [x] 基于 `eth_call` 的预执行模拟器
- [x] ERC20/ERC721/ERC1155 状态变更提取 (ΔS)
- [x] 配置系统（文件/环境变量）
- [x] CLI 工具（eval / batch / init-dataset）
- [x] 批量评估框架与度量指标
- [x] 基础单元测试（28+ 用例）
- [x] Fail-open 策略

### 主要缺口
- 缺少真实攻击交易数据集
- 缺少集成测试（依赖真实 RPC）
- 路径复杂度不变式 (I4) 默认仅标记，未拒绝
- 无多链验证
- 无监控/可观测性方案

## 阶段规划

### Phase 1 — 验证与数据（v0.2.0）
**目标**：用真实链上数据证明四项不变式的有效性

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 1.1 构建攻击数据集 | 收集 50+ 条历史攻击交易（rug-pull、Approval 滥用、貌似无害的 drain 操作），按攻击类型打标 | P0 |
| 1.2 构建良性基线集 | 覆盖主流 DeFi 协议（Uniswap、Aave、Compound、NFT 市场）的正常交易 200+ 条 | P0 |
| 1.3 集成测试框架 | 引入 pytest-vcr 或 httpx mock，录制真实 RPC 响应，使 CI 可离线运行 | P1 |
| 1.4 度量报告 | 输出 TPR / FPR / Fail-open 率，作为后续调参基准 | P1 |
| 1.5 参数调优文档 | 记录 `unlimited_allowance_threshold`、`max_expected_token_count` 等参数的推荐范围与调整方法 | P2 |

**交付物**：`data/attack_dataset.json`、`data/benign_dataset.json`、集成测试套件、度量基准报告

---

### Phase 2 — 核心能力增强（v0.3.0）
**目标**：提升检测精度，扩大攻击覆盖面

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 2.1 I4 路径不变式强化 | 在数据验证基础上评估是否默认开启拒绝；完善 call trace 解析（处理 STATICCALL、CREATE2） | P0 |
| 2.2 Permit / EIP-2612 检测 | 将离线签名授权（`permit`、Permit2）纳入 I2 权限不变式 | P0 |
| 2.3 批准目标地址校验 | I2 增强：检测授权接收方是否为已知合约/白名单，提升精度 | P1 |
| 2.4 ETH 价值异常检测 | 在 I1/I3 中加入 ETH 发送金额与历史行为的比较逻辑 | P1 |
| 2.5 多调用聚合解包 | 对 `multicall`、`execute`、`aggregate` 等聚合调用进行子调用展开后逐项分析 | P2 |
| 2.6 接受/拒绝理由语言化 | `GateDecision.violations` 生成人类可读的拒绝说明，便于钱包 UI 展示 | P2 |

**交付物**：增强版不变式引擎、Permit 检测模块、更新的测试用例

---

### Phase 3 — 生产加固（v0.5.0）
**目标**：使系统可嵌入真实钱包或安全工具

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 3.1 错误恢复与重试 | 按指数退避重试 RPC 失败；区分网络错误与无效响应 | P0 |
| 3.2 延迟 SLA | 目标单笔交易 p99 < 500ms；对慢路径（trace 获取）设置超时降级 | P0 |
| 3.3 异步批处理优化 | 利用 RPC batch call 并发获取多笔交易，提升批量评估吞吐 | P1 |
| 3.4 结构化日志 | 引入 structlog 或标准 JSON 日志格式，覆盖 gate 决策链路的关键节点 | P1 |
| 3.5 健康检查端点 | 暴露 `/health`（HTTP）或 CLI 诊断命令，检测 RPC 连通性与配置有效性 | P2 |
| 3.6 类型标注完善 | 补全 mypy strict 模式下所有 typing，CI 加入类型检查 | P2 |

**交付物**：生产就绪的核心库、性能基准报告、CI/CD 流水线

---

### Phase 4 — 多链与生态集成（v0.8.0）
**目标**：支持主流 EVM 链，提供 SDK 集成接口

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 4.1 多链配置 | 区分 Ethereum / Polygon / BSC / Arbitrum / Optimism 的链参数（chainId、原生 token、RPC） | P0 |
| 4.2 链特定常数扩展 | 补充各链主流 DEX/桥接合约的函数选择器和事件 topic | P1 |
| 4.3 Python SDK 封装 | 发布 `pip install wallet-invariant`，提供简洁 API：`gate.check(tx_dict) -> Decision` | P0 |
| 4.4 JSON-RPC 服务模式 | 可选 HTTP 服务端，供非 Python 钱包（JS/Go/Rust）通过 REST 调用 | P1 |
| 4.5 MetaMask Snap 概念验证 | 探索将 gate 逻辑移植到 MetaMask Snap，实现浏览器内预检 | P2 |
| 4.6 钱包集成文档 | 为不同宿主环境（extension、mobile、硬件钱包中间件）编写集成指南 | P2 |

**交付物**：多链支持、PyPI 发布包、JSON-RPC 服务、集成指南

### Phase 5 — 研究与生态（v1.0.0）
**目标**：学术发表、开源社区建设、标准化贡献

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 5.1 论文撰写 | 整理四项不变式形式化定义、实验数据、对比相关工作，投递学术会议（CCS/IEEE S&P/USENIX） | P0 |
| 5.2 开放基准数据集 | 整理攻击/良性数据集，以 MIT 许可开源发布，建立社区贡献机制 | P1 |

| 5.4 ERC 草案贡献 | 探索将"执行不变式"纳入 Ethereum ERC 标准（钱包安全扩展方向） | P2 |
| 5.5 审计报告 | 邀请外部安全团队对 gate 决策逻辑进行审计 | P2 |

**交付物**：论文草稿、开源数据集、对抗测试报告、v1.0.0 正式版

## 风险与注意事项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| RPC 节点不支持 `debug_traceTransaction` | I4 路径分析降级为标记模式 | 已有 fail-open 兜底；Phase 3 增加降级提示 |
| 高 FPR 阻断正常交易 | 用户体验下降，产品不可用 | Phase 1 先建基准，Phase 2 调参后再考虑生产部署 |
| 攻击者针对不变式构造绕过 | 检测失效 | Phase 5.3 主动对抗测试；不变式设计保持保守 |
| 多链差异导致分类器误判 | 跨链误报/漏报 | Phase 4.2 独立维护链特定常数 |
| 依赖 `web3.py` 版本升级 | API 破坏性变更 | 锁定 `>=6.0.0,<7.0.0`，定期评估升级 |
